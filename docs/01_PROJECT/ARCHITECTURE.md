# Project Architecture

## Purpose

Defines the lightweight MVP architecture for the internal Pharmacy CRM. This document is the source of truth for where code, documentation, tests, scripts, and configuration should live.

## Root Folder Responsibilities

| Folder | Responsibility |
| --- | --- |
| `docs/` | Product, architecture, database, and module documentation for the MVP. |
| `src/` | Application source code using Next.js App Router, feature-based architecture, and clean boundaries. |
| `prisma/` | Prisma schema, database migrations, and seed entrypoints. |
| `public/` | Static assets served directly by Next.js. |
| `tests/` | Test workspace for future checks. Keep it minimal. |
| `scripts/` | Operational scripts for database setup, backups, seeds, and migrations. |
| `.github/` | GitHub workflows and repository automation. |
| `.vscode/` | Workspace editor settings. |

## Source Architecture

```text
src/
  app/
  actions/
  components/
  features/
  hooks/
  lib/
  store/
  types/
  utils/
  constants/
  validations/
```

| Folder | Responsibility |
| --- | --- |
| `app/` | Next.js App Router routes, layouts, pages, route groups, loading states, and error boundaries. |
| `actions/` | Shared Server Actions that are not owned by one feature. Prefer feature-level actions when possible. |
| `components/` | Reusable UI components used across features. Feature-specific components stay inside their feature. |
| `features/` | Business modules organized by the MVP domains only. |
| `hooks/` | Shared React hooks used by multiple features. Feature-specific hooks stay inside their feature. |
| `lib/` | Infrastructure utilities such as Auth.js config, Prisma client, and permission helpers. |
| `store/` | Zustand stores for client-only UI state. Server data should not live here. |
| `types/` | Shared TypeScript types not generated by Prisma and not owned by one feature. |
| `utils/` | Small pure helpers used across the app. |
| `constants/` | Stable constants such as permissions, stock movement types, statuses, and routes. |
| `validations/` | Zod schemas for forms, Server Actions, and environment variables. |

## Feature-Based Structure

```text
src/features/
  auth/
  users/
  dashboard/
  products/
  parties/
  stock/
  ledger/
  expiry-alerts/
  reports/
```

Each feature should follow the same internal structure:

```text
src/features/products/
  components/
  actions/
  hooks/
  schemas/
  types/
  utils/
  constants/
  services/
```

| Feature Folder | Why It Exists |
| --- | --- |
| `components/` | UI pieces used only by this feature. |
| `actions/` | Feature-owned Server Actions for mutations and server-side commands. |
| `hooks/` | Feature-specific client hooks. |
| `schemas/` | Zod schemas for forms, filters, and action inputs. |
| `types/` | Feature-specific TypeScript types. |
| `utils/` | Small helpers that should not leak outside the feature. |
| `constants/` | Feature-specific statuses, options, and labels. |
| `services/` | Business logic and database coordination owned by the feature. |

### Routing Conventions

- Use route groups such as `(auth)` and `(app)` to separate layout concerns without changing URLs.
- Keep route files thin.
- Place mutation logic in feature actions or services, not in page components.
- Use nested routes only when the MVP truly needs them.
- Use `loading.tsx`, `error.tsx`, and `not-found.tsx` where they help.

| File | Responsibility |
| --- | --- |

| `prisma.ts` | Singleton Prisma client setup. |

### Soft Delete Policy

- Prefer deactivation or archival for entities that already have historical references.
- Physical deletion should be reserved only for records with no downstream references and no audit value.
- Products, batches, and parties should keep their history visible after deactivation.
- User records should remain in the audit trail even when inactive.

## Zustand Store Organization

```text
src/store/
  ui-store.ts
  sidebar-store.ts
  theme-store.ts
```

| Store | Responsibility |
| --- | --- |
| `ui-store.ts` | Global UI state such as dialogs and temporary interface flags. |
| `sidebar-store.ts` | Sidebar open/collapsed state and responsive navigation preferences. |
| `theme-store.ts` | Theme preference if not fully handled by a theme provider. |

### Do Not Store In Zustand

- Server data from Prisma.
- Auth sessions or permission truth.
- Stock quantities.
- Balances.
- Anything that must survive refresh without a persistence policy.


| Script | Responsibility |
| --- | --- |

| `migrate.ts` | Applies database migrations in a controlled deployment workflow. |

## Architecture Principles

- Use feature ownership for the MVP modules only.
- Use shared folders only for genuinely shared code.
- Keep domain rules out of React components.
- Validate all user input at server boundaries.
- Use Prisma transactions for stock and balance mutations.
- Keep Zustand for UI state only.
- Prefer Server Components for read-heavy pages.
- Avoid speculative abstractions.

## Naming and Scalability Best Practices

| Topic | Recommendation |
| --- | --- |
| Folder naming | Use kebab-case for route folders and lowercase names for domain folders. |
| File naming | Use kebab-case for non-component files and PascalCase for React components. |
| Components | Keep feature-specific components inside features; promote to shared only after reuse. |
| Imports | Use path aliases such as `@/features/products` and avoid deep fragile relative paths. |
| Aliases | Recommended aliases: `@/*`, `@/components/*`, `@/features/*`, `@/lib/*`, `@/types/*`. |
| Feature isolation | Features may depend on shared infrastructure, but should avoid importing internals from sibling features. |
| Clean code | Keep functions small, name business operations clearly, and avoid generic service buckets. |
| Reusable components | Build reusable components around real repetition, not imagined future needs. |
| Scalability | Add boundaries when the code demands them; do not create empty abstraction layers. |
